# Peer67 v2 — Anonymous Ephemeral Agent Messaging

## Overview

Peer67 is an open-source, encrypted, ephemeral messaging protocol for AI agents. Humans communicate through their Claude sessions — messages are end-to-end encrypted, stored on an anonymous relay for up to 24 hours, then deleted. The relay never knows who is communicating.

**Core properties:**
- **Zero metadata** — the relay stores opaque blobs at hashed addresses. No users, no accounts, no sender/recipient fields.
- **Ephemeral** — all messages auto-delete after 24 hours. No permanent storage anywhere on the server.
- **Invisible crypto** — users say "message Dana." Claude handles key exchange, encryption, and routing behind the scenes.
- **Shared identity** — connection data persists at `~/.peer67/` so all Claude sessions on a machine share the same contacts and keys.
- **Open protocol** — anyone can run a relay. The spec is public. Non-Claude agents can implement it.

## Architecture

```
Human A                                          Human B
   |                                                |
   v                                                v
Claude A                                       Claude B
   |                                                |
   v                                                v
~/.peer67/store.enc                      ~/.peer67/store.enc
(connections, keys)                      (connections, keys)
   |                                                |
   |  PUT /d/{mailbox} (encrypted blob)             |
   |  GET /d/{mailbox} (fetch blobs)                |
   |                                                |
   +----->-------+                +-------<---------+
                 |                |
                 v                v
          +----------------------------+
          |        Dead Drop Relay     |
          |                            |
          |  mailbox_id -> [blobs]     |
          |  24h TTL auto-delete       |
          |  No auth. No users.        |
          |  No logs. No metadata.     |
          +----------------------------+
```

## Components

### 1. Dead Drop Relay

A minimal HTTP server. Its only job is to hold encrypted blobs at addresses and delete them after 24 hours.

**Technology:** Node.js + Express (or Fastify) + Redis for TTL storage. Under 200 lines of server code.

**Endpoints:**

| Method | Path | Purpose |
|--------|------|---------|
| `PUT /d/{mailbox_id}` | Append an encrypted blob to a mailbox | 
| `GET /d/{mailbox_id}?after={timestamp}` | Fetch blobs newer than timestamp |
| `DELETE /d/{mailbox_id}/{blob_id}` | Delete a consumed blob |
| `GET /health` | Liveness check |

**Constraints:**
- Mailbox IDs are 64-character hex strings (256-bit).
- Max blob size: 64 KB.
- Max blobs per mailbox: 1000.
- Rate limit: 60 requests/minute per IP.
- All blobs auto-delete after 24 hours (Redis TTL or scheduled cleanup).
- No authentication. No logging of mailbox IDs or client IPs beyond rate limiting.
- CORS enabled for potential future web clients.

**Response format:**

```json
// PUT /d/{mailbox_id}
// Body: { "blob": "<base64 encrypted data>" }
// Response:
{ "id": "blob_uuid", "expires_at": "2026-04-01T12:00:00Z" }

// GET /d/{mailbox_id}?after=0
// Response:
{ "blobs": [
    { "id": "blob_uuid", "blob": "<base64>", "ts": 1711882800 }
  ]
}

// DELETE /d/{mailbox_id}/{blob_id}
// Response:
{ "ok": true }
```

**Abuse prevention (no auth needed):**
- Mailbox IDs are 256-bit — unguessable without the shared secret.
- IP-based rate limiting prevents flooding.
- 24h TTL means abuse is self-cleaning.
- 64KB max prevents the relay from being used as file storage.
- Optional future enhancement: proof-of-work on writes.

### 2. Local Identity Store (`~/.peer67/`)

All connection data lives on the user's machine. Claude reads it on session start.

**Files:**

```
~/.peer67/
  store.json          # Connections, encrypted at rest
  config.json         # Settings (default relay URL, poll interval)
```

**`store.json` structure (decrypted):**

```json
{
  "version": 1,
  "identity": {
    "name": "Suede",
    "created_at": "2026-03-31T..."
  },
  "connections": {
    "dana": {
      "display_name": "Dana",
      "mailbox_id": "a8f3...64 hex chars",
      "shared_key": "b7e2...64 hex chars",
      "relay_url": "https://relay.peer67.com",
      "created_at": "2026-03-31T...",
      "last_seen": "2026-03-31T..."
    }
  },
  "pending": {
    "outgoing": {
      "connection_code": "p67_...",
      "name": "Yossi",
      "private_key": "...",
      "created_at": "2026-03-31T..."
    }
  }
}
```

**Encryption at rest:** The store is encrypted with AES-256-GCM using a key derived from a machine-specific fingerprint (`hostname + username + machine-id`). This prevents casual file reads but is not a strong security boundary — the threat model assumes the local machine is trusted. Users who want stronger protection can set a master passphrase in `config.json` (which overrides the machine fingerprint).

**`config.json` structure:**

```json
{
  "default_relay": "https://relay.peer67.com",
  "poll_interval_seconds": 10,
  "master_passphrase_enabled": false
}
```

### 3. MCP Server (Claude Skill)

A single MCP server that provides tools to Claude. Replaces the Python scripts + SKILL.md from v1.

**Technology:** TypeScript, `@modelcontextprotocol/sdk`, runs as a subprocess of Claude Code.

**Tools exposed to Claude:**

| Tool | Purpose |
|------|---------|
| `peer67_connect` | Generate a connection code, or accept one |
| `peer67_send` | Encrypt and send a message to a contact |
| `peer67_inbox` | Check for and decrypt new messages |
| `peer67_contacts` | List active connections |
| `peer67_disconnect` | Remove a connection |

**Tool schemas:**

```typescript
// peer67_connect
{
  action: "create" | "accept",
  name: string,                    // Human-readable name for this contact
  code?: string,                   // Required for "accept"
  relay_url?: string,              // Override default relay for "create"
}
// Returns: { code: string } for "create", { connected: true } for "accept"

// peer67_send
{
  to: string,                      // Contact name
  message: string,                 // Plaintext (encrypted before sending)
}
// Returns: { sent: true, expires_at: string }

// peer67_inbox
{
  from?: string,                   // Filter by contact name (optional)
}
// Returns: { messages: [{ from, body, timestamp }] }

// peer67_contacts
{}
// Returns: { contacts: [{ name, relay_url, connected_at, last_seen }] }

// peer67_disconnect
{
  name: string,
}
// Returns: { disconnected: true }
```

**Behavioral rules (embedded in MCP server instructions):**
- Never send a message without user confirmation. Show draft, wait for "yes"/"send it."
- Never store message content in Claude's memory. Contacts and connection status only.
- On session start, read `~/.peer67/store.json` and report any unread messages.
- Present messages conversationally, never as raw JSON.

**Background polling:**
The MCP server runs a background loop (configurable interval, default 10s) that checks all mailboxes. When new messages arrive, it pushes a notification into the Claude session via `notifications/claude/channel`.

### 4. Connection Protocol

The handshake uses X25519 key agreement so neither party needs to share a secret in advance. The connection code carries the initiator's public key.

**Key exchange steps:**

```
1. INITIATOR (Suede's Claude):
   - Generate X25519 keypair: (private_a, public_a)
   - Create connection code: base62(public_a || relay_url || nonce)
   - Store pending connection in ~/.peer67/store.json
   - Display code for user to share

2. ACCEPTOR (Dana's Claude):
   - Decode connection code → public_a, relay_url, nonce
   - Generate X25519 keypair: (private_b, public_b)
   - Perform key agreement: shared_secret = X25519(private_b, public_a)
   - Derive keys via HKDF-SHA256:
     - mailbox_a_to_b = HKDF(shared_secret, "peer67-mailbox-a2b")
     - mailbox_b_to_a = HKDF(shared_secret, "peer67-mailbox-b2a")
     - encrypt_key    = HKDF(shared_secret, "peer67-encrypt")
   - Store connection in ~/.peer67/store.json
   - Send public_b to mailbox_a_to_b (so initiator can complete)

3. INITIATOR (next poll):
   - Finds public_b in mailbox_a_to_b
   - Performs key agreement: shared_secret = X25519(private_a, public_b)
   - Derives same mailbox IDs and encryption key
   - Stores connection. Handshake complete.
```

**Why two mailboxes?** Each direction gets its own mailbox ID. This prevents the relay from even correlating that two blobs are part of the same conversation. Suede writes to `mailbox_a_to_b`, Dana reads from it. Dana writes to `mailbox_b_to_a`, Suede reads from it.

**Connection code format:**

```
p67_<base62 encoded payload>

Payload (binary):
  [32 bytes] X25519 public key
  [2 bytes]  relay URL length
  [N bytes]  relay URL (UTF-8)
  [16 bytes] random nonce
```

The `p67_` prefix makes codes recognizable. Example: `p67_7Kj3mX9pLqR2vN5wT8...`

### 5. Message Format

Each blob stored on the relay is an encrypted envelope:

```
Plaintext envelope (before encryption):
{
  "v": 1,                         // Protocol version
  "t": 1711882800,                // Unix timestamp (seconds)
  "b": "The deck is ready"        // Message body
}

Encrypted with AES-256-GCM:
  - Key: encrypt_key (from HKDF)
  - Nonce: 12 random bytes (prepended to ciphertext)
  - AAD: mailbox_id (binds ciphertext to destination)

Wire format (base64 of):
  [12 bytes] nonce
  [N bytes]  AES-256-GCM ciphertext + auth tag
```

Using the mailbox ID as AAD (additional authenticated data) prevents an attacker from copying a blob from one mailbox to another.

## User Experience

### First time setup

```
User: set up peer67
Claude: What name should I use for you?
User: Suede
Claude: Done. You're set up as "Suede". To connect with someone, 
        say "connect me with [name]" and I'll give you a code to share.
```

### Connecting

```
User: connect me with Dana
Claude: Here's your connection code. Send it to Dana however you like:

        p67_7Kj3mX9pLqR2vN5wT8yB...

        Once Dana enters it in her Claude, you'll be connected.

---

[Dana's side]
User: connect with Suede
Claude: Paste the code Suede sent you.
User: p67_7Kj3mX9pLqR2vN5wT8yB...
Claude: Connected with Suede. You can now message each other.

---

[Back on Suede's side, next poll]
Claude: Dana accepted your connection. You're now connected.
```

### Sending

```
User: tell Dana the deck is ready
Claude: I'll send this to Dana:
        "The deck is ready"
        Send it?
User: yes
Claude: Sent. Message expires in 24 hours.
```

### Receiving

```
User: check messages
Claude: 1 new message:
        Dana (3 min ago): "Can you review the slides before 3pm?"
        
        Want to reply?
```

### Push (when MCP channel is active)

```
Claude: New message from Dana: "Meeting moved to Thursday."
        Want to reply?
```

## Relay Scalability

**Multiple relays, zero registration.** Anyone can deploy a relay — it's a stateless blob store. The connection code embeds the relay URL, so different connections can use different relays.

**Default relay:** `relay.peer67.com` (domain: `peer67.com`, owned) — hosted by the project, free tier.

**Custom relays:** Override via `config.json` or per-connection at code generation time.

**Future: DNS discovery.** `_peer67.example.com` TXT record could advertise a relay URL. Not needed for v1.

## Security Model

**What the relay sees:**
- Hex mailbox IDs (opaque, unlinkable to any identity)
- Encrypted blobs (opaque bytes)
- Client IP addresses (for rate limiting only, not logged)
- Request timestamps

**What the relay CANNOT see:**
- Who is communicating (no identity on either side)
- Message content (encrypted)
- Which mailbox pairs belong to the same conversation (separate IDs per direction)
- Whether a mailbox is being read or written by the same person or different people

**What the relay CANNOT do:**
- Correlate sender and recipient (it doesn't know they exist)
- Read messages (doesn't have keys)
- Forge messages (AAD binding + AES-GCM auth tag)
- Replay messages to a different mailbox (AAD mismatch)

**What would break security:**
- Compromised local machine (attacker reads `~/.peer67/store.json` — mitigated by encryption at rest)
- Compromised relay + traffic analysis (timing correlation of PUT/GET to same mailbox — mitigated by two separate mailbox IDs per connection)
- Man-in-the-middle during code exchange (attacker intercepts connection code — mitigated by sharing codes over a trusted channel)

**Future enhancement: safety numbers.** After connecting, both Claudes could display a short hash of the shared secret for users to verify out-of-band, like Signal safety numbers.

## Tech Stack

| Component | Technology |
|-----------|-----------|
| Relay server | Node.js + Fastify (minimal) |
| Blob storage | Redis (TTL keys) or SQLite (cleanup cron) |
| MCP server | TypeScript + @modelcontextprotocol/sdk |
| Crypto | libsodium (X25519, HKDF, AES-256-GCM) via `sodium-native` or `@noble/curves` + `@noble/ciphers` |
| Local store | Encrypted JSON file |
| Deployment | Railway / Fly.io / Docker |

**Dependency philosophy:** Minimal. The relay should have 3-4 dependencies. The MCP server should have 5-6. No ORMs, no heavy frameworks. Everything auditable.
